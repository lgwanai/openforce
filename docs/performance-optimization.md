# OpenForce 性能优化方案

## 问题分析

当前系统响应时间过长（约 100 秒），主要瓶颈：

1. **LLM 延迟高**：平均 10 秒/次调用
2. **ReAct 循环多**：户部需要多次 LLM 调用
3. **串行执行**：所有操作都是串行的

## 优化方案

### 方案 1：减少 LLM 调用次数（推荐）

**问题**：户部使用 ReAct 模式，每步都要调用 LLM 决策

**解决**：
1. 使用 **Plan-and-Execute** 模式替代 ReAct
2. 一次性规划好所有工具调用，然后批量执行

```python
# 当前：ReAct 模式（多次 LLM）
思考 → 调用工具 → 观察结果 → 思考 → 调用工具 → ...

# 优化后：Plan-and-Execute 模式（1次 LLM 规划 + 批量执行）
规划：[调用A, 调用B, 调用C] → 并行执行 → 汇总结果
```

**预期效果**：户部从 4 次 LLM 调用减少到 1-2 次

### 方案 2：并行执行工具调用

**问题**：工具调用是串行的

**解决**：对于独立的工具调用，使用并行执行

```python
# 例如：同时搜索多个数据源
import asyncio

async def parallel_search(query):
    tasks = [
        web_search(query),
        fetch_webpage(url1),
        fetch_webpage(url2)
    ]
    return await asyncio.gather(*tasks)
```

### 方案 3：使用更快的模型

**问题**：当前使用 kimi-k2.5，延迟约 10 秒

**解决**：
1. 使用更轻量的模型（如 glm-4-flash）做简单决策
2. 只在需要深度推理时使用大模型

| 模型 | 延迟 | 适用场景 |
|-----|------|---------|
| glm-4-flash | ~2s | 简单决策、工具选择 |
| minimax-m2.5 | ~5s | 中等复杂度 |
| kimi-k2.5 | ~10s | 深度推理 |

### 方案 4：缓存和预取

**解决**：
1. 缓存常用查询结果（天气、新闻等）
2. 预取可能需要的数据

### 方案 5：简化户部职责

**当前问题**：户部做了太多决策

**解决**：让中书省做更多规划，户部只执行

```
用户 → 中书省（规划查询步骤） → 户部（执行） → 中书省（汇总）
```

## 实施优先级

1. **高优先级**：方案 3（使用更快的模型）
2. **高优先级**：方案 1（减少 LLM 调用）
3. **中优先级**：方案 2（并行执行）
4. **低优先级**：方案 4（缓存）

## 预期效果

| 优化前 | 优化后 |
|-------|-------|
| ~100s | ~20s |
| 4次 LLM | 1-2次 LLM |
| 串行执行 | 并行执行 |
