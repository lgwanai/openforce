# OpenForce (三省六部多智能体系统)

## 📖 介绍
OpenForce 是一个具备弹性架构的分层多智能体系统。它摒弃了传统单体 Agent 或扁平化多 Agent 系统中常见的职责混乱与上下文超载问题，将国家治理的机制巧妙地融入了 Agent 的设计之中，通过层级验证、动态编排和严格的职责划分，实现对各类复杂任务的高效处理。系统不再强制所有任务都经过复杂的全链路编排，而是根据意图复杂度和所需能力动态选择执行路径：简单的直接回，少量工具调用直接做，复杂的深思熟虑并分层拆解。

## ✨ 项目亮点：借鉴千古智慧的治理机制
核心借鉴了中国千年老祖宗的智慧——**“三省六部制”**。
- **中书省 (The Brain - 决策与规划)**：负责接收用户指令，进行意图识别和顶层任务规划（Plan）。它拥有直接回复权，亲自持有操作系统级轻量工具，能够对简单任务进行“秒级”的 L0/L1 响应。
- **尚书省 (The Orchestrator - 编排与调度)**：负责资源调度、任务拆解与流程监控。它将顶层计划拆解为子计划，派发给“六部”执行，并维护宏观的 `Plan -> Execute -> Review` 状态机任务队列，是系统的中枢枢纽。
- **六部 (The Worker Pool - 执行工作组)**：动态的专业 Agent 资源池。尚书省可以根据任务需求，实例化多个同类 Agent，并挂载相应的 Skill。
  - **兵部 (Executor)**：通用执行、代码编写、复杂工具调用。
  - **户部 (Information Bureau)**：自主研究员（Research Agent），负责不确定性信息的获取、深度调研与确证。
  - **工部 (Environment)**：负责沙箱环境维护、依赖安装。
  - **刑部 (Reviewer)**：负责代码审查、功能测试与主观/完整性验收。
  - **吏部 (Skill & Tool Manager)**：技能与外部工具库（MCP/Skill）的守护者与分发者。
  - **礼部 (SOP)**：负责流程标准化、文档生成。
  - **都察院 (Policy & Security)**：负责权限模型、敏感数据策略、审计与合规检查。

## 🤔 为什么要这个 Agents？
现代 AI 任务正变得越来越复杂。传统的 Agent 架构往往因为缺乏层级和有效验证，容易陷入“幻觉雪崩”、死循环或不可控的资源消耗。我们需要 OpenForce，因为：
1. **防失忆与上下文撕裂**：通过统一的 Context Store、全局游标 (Cursor) 以及动态指针 (Pointer) 模式，有效防止超长任务中的大模型失忆，且不因“快速路径”丢失上下文。
2. **防幻觉与自欺欺人**：引入**层层目标验证闭环（Plan & ReAct）与硬性断言 (Hard Assertions)**，底层不仅依靠大模型主观判断，更结合沙箱退出码等客观事实，未达标绝不向上提交。
3. **极高的系统自洽性**：反硬编码（Anti-Hardcoding），全部基于语义路由与动态 ReAct 循环，不做 `if-else` 的场景枚举。

## 🎯 核心工作流
1. **消息接入与意图路由**：用户通过 CLI/IM/API 输入指令，中书省基于语义进行智能分流（Chat、基础工具闭环、知识问答、复杂任务）。
2. **顶层规划与预算装配**：面对复杂任务，中书省通过引导式澄清锁定目标，生成带有验收标准（Acceptance Criteria）的顶层 Plan，并冻结全局预算。
3. **动态编排与并发派发**：尚书省接管 Plan，拆解为带有依赖关系的 DAG，并 Spawn 出具体的“六部”子 Agent。
4. **智能 ReAct 执行与自验证**：底层子 Agent 进行“思考-分解-执行-检验”的多轮尝试。结合沙箱日志进行自主修正，如果陷入死循环则触发熔断退避。
5. **汇总与防爆破压缩**：当内容过大时，底层强制触发内容压缩（Summary + File Pointer）。尚书省汇总后交由中书省进行顶层验收，最终交付用户。

## 💡 高效、周全的设计逻辑 
OpenForce 在设计上极度追求工程的自洽性、健壮性与安全边界：
- **分级响应 (L0-L3)**：
  - **L0 简单回应**：中书省直接回复（<1s）。
  - **L1 基础操作**：中书省调用原生轻量工具（<3s）。
  - **L2 知识问答**：中书省直通户部深度调研（<10s）。
  - **L3 复杂任务**：中书省转尚书省动态编排。
- **带超时的并发屏障 (Timeout Barrier)**：摒弃传统的 `Promise.all`。并行子任务触发超时或死循环会强制释放屏障，返回部分状态唤醒尚书省重新规划，彻底杜绝系统死锁。
- **沙箱隔离与防逃逸**：不仅在字符串层面过滤绝对路径，更在底层实现“真实路径归一化 (realpath)”，严禁跟随符号链接 (Symlink)，防止 Prompt Injection 诱导越权。
- **人机协同与审批防伪 (Human-in-the-loop)**：高风险操作强制插入人工审批节点。审批采用包含 `task_id/action_hash/nonce` 的一次性签名令牌，彻底杜绝越权审批与消息重放攻击。执行前还会进行对象冻结快照比对，防止 TOCTOU 漂移。
- **副作用一致性 (Hybrid Outbox 模式)**：对外部写操作使用 `operation_id` 幂等键。长时调用（>5s）采用 `PREPARE -> SUBMIT -> Waiting -> COMMIT` 协议挂起，保证“至多一次副作用”。
- **单机单用户隔离与会话门控**：采用严格的 Session Gate 和 DB 原子操作控制并发。当处于“任务进行中”时，静默拦截干扰请求，防止上下文污染。
- **全局预算与熔断**：每个 Task 拥有 `time_budget`、`token_budget`。并行 Agent 必须切分锁定独立子预算，防止单个失控 Agent 耗尽全局资源导致其他线程“饿死”。

## 🛠️ 使用场景
- **深度知识调研**：遇到不确定的技术问题时，直接呼叫户部（Research Agent），自动规划搜索关键词，深度阅读并提炼总结，主动过滤冲突与过时信息。
- **复杂软件开发**：中书省接收需求后，尚书省协调工部（搭环境）、兵部（写代码）、刑部（跑测试）。兵部卡壳时可随时寻求户部帮助，全自动闭环。
- **系统运维与自动化控制**：挂载 MCP/Skill 之后，系统能自主分析日志、排查故障。并在执行任何可能引起变更的敏感命令前，由都察院拦截并发送授权卡片供人类确认。

## 📦 安装方式
*(注：项目正在高速迭代中，以下为基础环境配置示例)*
```bash

git clone https://github.com/your-username/openforce.git
cd openforce

# 2. 创建并激活虚拟环境
python3 -m venv venv
source venv/bin/activate

# 3. 安装依赖 (开发中)
pip install -r requirements.txt

cp config/models.yaml.example config/models.yaml
# 编辑 config/models.yaml，填入你的 OpenAI / Anthropic / DeepSeek 等 API Key

# 5. 启动系统
python main.py
```

## ❓ 常见问题 (FAQ)

**Q: OpenForce 支持哪些大模型？**
A: 系统采用多供应商、多模型并行配置策略。你可以为 Planner、Coder、Research 等不同角色独立指定主要模型与候选回退模型（Fallbacks）。例如：用 Claude 3.5 Sonnet 做尚书省的调度，用 GPT-4o 写代码，用 DeepSeek 做户部的深度推理。

**Q: 任务执行一半如果报错、超时或网络挂起了怎么办？**
A: 系统拥有完整的任务状态机和图快照恢复（Graph Snapshot）机制。底层 Worker 具有局部 ReAct 重试与指数退避机制；如果遇到致命错误或超时，会向上层抛出明确的 Error Report，由尚书省进行 Re-Plan。如果是外部 API 长时间挂起，会被放入 `WaitingExternalCall` 后台轮询，绝不阻塞主线程。

**Q: 如何保证 Agent 不会乱删我的文件或执行危险命令？**
A: OpenForce 引入了**都察院**角色和**安全与一致性协议**。所有高风险工具都会进行严格的参数校验和脱敏（Secret Redaction）。对于涉及资产变更或核心数据删除的操作，被定义为绝对禁区（永不豁免），系统会强制挂起并向用户发起一次性 Token 的授权审批，未授权绝不放行。
